# v0.5.0 — Showcase Stabilization

## Mission

Stabilize the visual experience as a shippable showcase: reliable first load,
clean defaults, reproducible review artifacts, updated docs, and no hidden
regressions from the v0.3/v0.4 visual work.

This is not an education-mode milestone. It is the “make the beautiful thing
safe to show” milestone.

Done = the app can be sent to someone as a live link, opens to the accepted look,
keeps frame/perf telemetry honest, has current docs, and has a repeatable way to
capture screenshots/video for future regressions.

## Product goal

By this point, the project should feel like a polished neural sculpture: a
procedural brain-shaped arena, organic curved neurons, visible soma pulses,
traveling impulses, subtle material polish, and smooth navigation. v0.5.0 makes
that stable enough to demo without handholding. It does not wait for deferred
soft region territories.

## Context

`architecture/build-and-deploy.md` owns build/test/deploy. `architecture/profiling.md`
owns the public HUD and metrics pipeline. `architecture/gpu-rendering.md` owns
visual pass order and shader behavior. Repo rules require gates before commit and
code+docs together.

This plan should not add large new features. It should harden what already
exists.

## Current verification status

As of `2026-06-08`, the code/build stabilization work that is possible in this
environment is complete:

- `cargo test -p brain-visualizer` passed.
- `cargo run -p brain-visualizer --example render_check` passed.
- `cargo run -p brain-visualizer --example morph_view` passed and refreshed the
  `/tmp/morph_*` review artifacts plus JSON stats files.
- `npm run typecheck`, `npm test`, and `npm run build` passed.
- `npm run preview` served the built app with COOP/COEP headers, and Chromium
  booted the production build.
- The public HUD regression was fixed during stabilization: the built app again
  shows the profiler-driven corner HUD instead of silently dropping it.
- Clean first-load/default/reset behavior and hidden review preset payloads are
  consistent with `DEFAULT_CONFIG`, `DEFAULT_SETTINGS`,
  `DEFAULT_MORPH_CONFIG`, and the accepted-default preset contract.

Still blocked here:

- this environment exposes `navigator.gpu` in Chromium but no usable adapter via
  `requestAdapter()`, so a real browser WebGPU beauty pass is still manual;
- Adam's subjective visual acceptance is still required before marking the look
  shipped.

The accepted look is defined by
[`accepted-visual-defaults-manifest.md`](accepted-visual-defaults-manifest.md).
v0.5.0 verifies that manifest against production build behavior; it should not
re-tune defaults unless verification exposes a bug or an explicitly reopened
v0.4.0 acceptance issue.

## Out of scope

- Education lessons.
- New simulation model features.
- New renderer architecture.
- CPU backend revival.

## Stabilization checklist

### First-load experience

- Empty localStorage opens to the accepted default look.
- Brain silhouette is visible before interaction.
- Activity naturally starts without a scripted intro.
- Hover stimulation still feels like a light perturbation.
- Right-drag pan, left-drag orbit, and wheel zoom are discoverable enough by
  behavior even without a settings page.

### Visual robustness

- Bloom does not wash out low-tier hardware settings.
- Morphology pass does not exceed segment budgets.
- Pulses remain legible at normal and slow speeds.
- Material pass does not produce flicker or camera crawl.
- Brain shell remains readable with morphology and far glow enabled.
- The cortical-shell placement still avoids a glowing solid-volume look.

### Performance honesty

- Public corner HUD still updates normally.
- No new GPU readback was added to the rAF loop.
- No new per-frame allocation was added to the render loop.
- The default tier remains within the intended target on representative hardware.
- Any llvmpipe/offscreen numbers stay out of architecture docs as benchmark
  claims.

### Docs and plans

- Architecture docs describe current state, not version history.
- Decision docs preserve accepted choices only.
- `future_roadmap.md` no longer lists completed work as deferred.
- `visual-acceptance-contract.md` reflects the final accepted artifact set.
- `accepted-visual-defaults-manifest.md` contains the exact accepted settings,
  hidden preset payloads, camera presets, and final artifact ledger.
- This plan and completed v0.3/v0.4 plans can be marked done or archived per the
  repo's plan lifecycle.

The files themselves do not need to be committed unless the repo already stores
review artifacts. The capture checklist is enough if artifacts are large, but the
artifact JSON/path list must be recorded in the accepted defaults manifest and
final handoff.

## Implementation steps

1. Start from clean localStorage and production-equivalent build.
2. Load the accepted defaults manifest and verify first-load/default,
   performance, and hero hidden presets match it.
3. Run the test gates.
4. Run visual harnesses and manual browser review.
5. Fix any default-setting or localStorage migration problems.
6. Clean up docs touched by v0.3/v0.4 work.
7. Update `future_roadmap.md` deferred/rejected tables.
8. Capture final review media.
9. Do a final pass for stale comments, tombstoned settings, and misleading old
   plan references.
10. Leave soft spatial region territories in backlog unless deliberately
    reopened.

## Exit gate

- `cargo test -p brain-visualizer` passes.
- `npm run typecheck` passes.
- `npm test` passes.
- Production build passes.
- Production preview loads with WebGPU path on supported hardware.
- First-load default look is accepted.
- Reset-to-defaults returns to the accepted look.
- Accepted defaults manifest reproduces first-load and hidden dev-panel preset
  configs exactly.
- Final artifacts satisfy [`visual-acceptance-contract.md`](visual-acceptance-contract.md).
- Public HUD still reports FPS/N/backend/synaptic events as intended.
- Architecture and decision docs are current.
- No completed v0.3/v0.4 item remains listed as deferred future work.

## Discipline rules

- Do not sneak new product features into stabilization.
- Do not add a public dashboard UI.
- Do not promise hardware performance based on software-emulated llvmpipe.
- Do not commit generated build artifacts.
- Do not leave docs saying “previously” / “v0.3 added” in architecture files.
