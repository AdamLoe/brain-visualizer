# v0.3.4 — Right-Click Pan & Camera Target

## Mission

Add screen-space camera panning with right-click drag while preserving the simple
watch-and-perturb interaction model.

Current state: left-drag/orbit and scroll/zoom are supported. Hover stimulation
runs only when no drag is occurring. The camera orbits around the origin and has
no target/pan offset.

Done = left-drag still orbits, right-drag pans left/right/up/down in world
space, scroll still zooms, hover stimulation still works when no button is held,
and the backend/simulation are untouched.

## Product goal

The user should be able to slide the brain around the screen without fighting
the orbit camera. This is especially useful once the brain is no longer a sphere
and close-up neurons fill the frame.

## Context

`architecture/web-frontend.md` owns the rAF loop, input events, pending flags,
and camera. `web/src/render/camera.ts → Camera` produces the MVP matrix, camera
right/up vectors, and unprojection rays.

`decisions/interaction.md` currently describes the shallow input model:
hover=stimulate, drag=orbit, scroll=zoom, click does nothing. This plan keeps the
same spirit and updates the input detail to: left-drag=orbit, right-drag or
Shift-left-drag=pan, scroll/pinch=zoom, hover with no button=stimulate, click
still does nothing.

The browser event handlers must not call into the wasm backend directly. They
only update frontend state/pending flags; the rAF loop remains the single owner
of backend calls.

## In scope

- Add `target` / `panOffset` state to the camera.
- Orbit around `target` instead of always orbiting around the origin.
- Add `Camera.pan(dx_px, dy_px, viewport_width, viewport_height)`.
- Use camera right/up vectors to translate the target in screen space.
- Add right-button drag handling in the canvas pointer handlers.
- Disable the browser context menu over the canvas.
- Add Shift-left-drag as the desktop fallback if it is cheap and does not
  disturb orbit handling.
- Keep hover stimulation disabled while any button drag is active.
- Preserve mobile behavior unless a safe two-finger pan falls out naturally.
- Add reset/recenter behavior if needed to avoid losing the brain offscreen.

## Out of scope

- Click-to-inspect.
- Object picking or GPU id buffer.
- Touch gesture redesign.
- WASM/backend API changes.
- Simulation-coordinate changes.
- Camera path animation or cinematic mode.

## Camera model

Current camera can be treated as spherical offset plus distance. Add a target:

```text
eye = target + spherical_offset(azimuth, elevation, distance)
view = look_at(eye, target, world_up)
```

Pan translates `target` along the camera's screen-plane basis:

```text
world_per_pixel = 2 * distance * tan(fov_y / 2) / viewport_height
pan_delta = -camera_right * dx_px * world_per_pixel
          +  camera_up    * dy_px * world_per_pixel
target += pan_delta
```

Sign may need tuning by hand. The user should feel like dragging the scene, not
like dragging the camera.

## Input behavior

Desktop:

| Input | Behavior |
|---|---|
| left drag | orbit |
| right drag | pan |
| wheel | zoom |
| hover, no buttons | stimulation |
| click release | no action |
| context menu on canvas | suppressed |

Desktop fallback:

| Input | Behavior |
|---|---|
| Shift + left drag | pan, if implemented without compromising orbit |

Mobile remains as-is for this plan:

| Input | Behavior |
|---|---|
| one finger | orbit |
| two fingers | pinch zoom |
| stimulation | disabled |

## Implementation steps

1. Add `target: Vec3` or equivalent to `web/src/render/camera.ts → Camera`.
2. Update MVP/view computation to look at `target`.
3. Ensure `cameraRight()`, `cameraUp()`, and `eye()` still return correct values
   after the target offset.
4. Add `panByPixels(dx, dy, viewport)` to `Camera`.
5. Update pointer state tracking in `web/src/main.ts` / camera handlers to
   distinguish left-button orbit from right-button pan.
6. Treat `PointerEvent.buttons & 1` as orbit and `PointerEvent.buttons & 2` as
   pan. If both are held, prefer the currently active mode or ignore until
   buttons settle.
7. Call `preventDefault()` for right-drag and suppress `contextmenu` on canvas.
8. Ensure hover stimulation receives `false`/no pending stimulation while panning.
9. Add a recenter action if the camera can be panned too far. Keep it minimal:
   keyboard `R`, double-click background, or an internal clamp are all acceptable
   if they do not clutter the public UI.

## Review checklist

- Right-drag feels like sliding the brain in the screen plane.
- Left-drag orbit behavior is unchanged.
- Wheel zoom still zooms toward the current target.
- Hover stimulation does not trigger while panning.
- Browser context menu does not appear over the canvas.
- Ray/unprojection math still feels correct after panning.
- Mobile behavior is not regressed.
- The shared visual acceptance contract's pan check passes.

## Exit gate

- No wasm/backend API changes.
- No new per-frame allocations in the rAF loop.
- TypeScript typecheck passes.
- Existing pointer/orbit tests pass or are updated.
- Manual browser check: left orbit, right pan, wheel zoom, hover stimulation,
  mobile/touch sanity.
- `npm run typecheck` passes.
- `npm test` passes.
- `cargo test -p brain-visualizer` still passes if the full repo gate is run.

## Discipline rules

- Do not make right-click select or inspect neurons.
- Do not route pointer handlers directly into `WasmGpuBackend`.
- Do not add a public settings panel for camera behavior.
- Do not break hover stimulation's “no button held” rule.
